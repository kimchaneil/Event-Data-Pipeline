# AWS 아키텍처 설계

이 프로젝트의 파이프라인을 AWS에서 운영한다고 가정하면, 

애플리케이션 실행 환경으로 ECS, 이벤트 저장소로 RDS for PostgreSQL, 시각화 결과 파일 저장소로 S3, 로그 및 상태 모니터링을 위해 CloudWatch를 사용할 수 있습니다. 

사용자는 ECS에 배포된 이벤트 생성 애플리케이션에 요청을 보내고, 애플리케이션은 생성한 page_view, purchase 이벤트를 RDS에 저장합니다. 

이후 시각화 로직은 RDS의 집계 결과를 조회하여 차트를 생성하고, 생성된 PNG 파일을 S3에 저장합니다. 

애플리케이션 로그와 실행 상태는 CloudWatch로 전송하여 모니터링합니다.

### 선택한 AWS 서비스의 역할 차이

**ECS** 는 애플리케이션 코드를 실행하는 컴퓨팅 환경입니다. 이 프로젝트처럼 Docker 기반으로 구성된 API, 프론트엔드, 시각화 작업을 컨테이너 단위로 운영하기에 적합합니다.

**RDS for PostgreSQL**은 이벤트 로그를 구조화해서 저장하는 데이터베이스 역할을 합니다. 조회, 구매, 전환율, 시간대 분석 같은 SQL 집계를 수행할 수 있습니다.

**S3**는 데이터베이스처럼 레코드를 저장하는 곳이 아니라, 생성된 차트 PNG 같은 결과 파일을 저장하는 객체 스토리지입니다.

**CloudWatch**는 애플리케이션을 실행하는 서비스가 아니라, 실행 중인 애플리케이션의 로그와 상태를 수집하고 모니터링하는 서비스입니다.

### AWS 서비스를 선택한 이유

**ECS**를 선택한 이유는 현재 프로젝트가 Docker Compose 기반으로 구성되어 있어, AWS에서도 컨테이너 실행 환경으로 옮기는 것이 가장 자연스럽다고 판단했기 때문입니다.

**RDS for PostgreSQL**은 이벤트 데이터를 컬럼 단위로 저장하고 SQL 집계를 수행하기에 적합해 선택했습니다.

**S3**는 시각화 결과를 이미지 파일로 저장해야 하는 요구사항을 비용 효율적으로 만족할 수 있어 선택했습니다.

**CloudWatch**는 운영 환경에서 로그 확인과 장애 대응이 필요하다고 판단하여 포함했습니다.

## 가장 고민한 부분

두가지 고민한 부분 중 하나는 **시각화 결과를 S3에 파일로 저장할지, BI 도구를 사용해 바로 대시보드로 보여줄지**에 대한 점이었습니다. 

BI 도구는 탐색성과 분석 편의성이 높지만 비용이 더 커질 수 있고, 현재 과제 상황을 회사에서 운영하는 이벤트 로그를 발생 상황을 가정하여 분석하기 위했기 때문에 S3를 선택하였습니다.

그래서 이번 설계에서는 비용 효율성과 현재 구현 방식의 일관성을 고려하여, 시각화 결과를 PNG 파일로 생성한 뒤 S3에 저장하는 구조를 선택했습니다.

두번째 고민은,  **애플리케이션 실행 환경으로 EC2와 ECS 중 어떤 서비스를 선택할지**였습니다.

EC2는 가장 직관적인 컴퓨팅 자원이어서 설명이 단순하다는 장점이 있었지만, 현재 프로젝트가 docker compose 기반으로 구성되어 있고 API, 프론트엔드, 시각화 작업이 컨테이너 단위로 나뉘어 있다는 점을 고려하면 **ECS**가 더 자연스럽다고 판단했습니다.

 따라서 이번 설계에서는 현재 구현 구조와의 연결성, 컨테이너 기반 운영의 일관성을 우선하여 **ECS**를 선택했습니다.
